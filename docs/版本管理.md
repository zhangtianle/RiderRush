## 一、版本号规范

### 1.1 版本号格式

采用语义化版本号（Semantic Versioning）：

```
格式：MAJOR.MINOR.PATCH[-STAGE]

示例：
├── v0.1.0-alpha   # MVP开发阶段
├── v0.2.0-beta    # 第一轮迭代测试
├── v1.0.0         # 正式发布版本
├── v1.1.0         # 功能更新版本
└── v1.0.1         # Bug修复版本
```

### 1.2 版本号规则

| 版本类型 | 变更规则 | 示例 |
|----------|----------|------|
| **MAJOR** | 架构重大变更、不兼容更新 | v1.0.0 → v2.0.0 |
| **MINOR** | 新功能添加、向后兼容 | v1.0.0 → v1.1.0 |
| **PATCH** | Bug修复、小优化 | v1.0.0 → v1.0.1 |
| **STAGE** | 开发阶段标记 | v0.1.0-alpha |

### 1.3 开发阶段标记

| 阶段标记 | 说明 | 使用场景 |
|----------|------|----------|
| `-alpha` | 开发阶段 | MVP开发期间 |
| `-beta` | 测试阶段 | 内测/公测期间 |
| `-rc` | 发布候选 | 正式发布前的最终测试 |
| 无标记 | 正式版本 | 已发布稳定版本 |

## 二、Git分支规范

### 2.1 分支结构

```
分支结构：

main (主分支)
  │
  ├── develop (开发分支)
  │     │
  │     ├── feature/core-classes (功能分支)
  │     ├── feature/level-system
  │     ├── feature/ui-scenes
  │     ├── feature/visual-pack
  │     ├── feature/ad-monetization
  │     │
  │     └── bugfix/collision-fix (修复分支)
  │     └── bugfix/performance-fix
  │
  └── release/v0.1.0 (发布分支)
  └── release/v1.0.0
```

### 2.2 分支命名规范

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| **主分支** | `main` | main |
| **开发分支** | `develop` | develop |
| **功能分支** | `feature/功能名` | feature/level-system |
| **修复分支** | `bugfix/问题描述` | bugfix/collision-fix |
| **发布分支** | `release/版本号` | release/v0.1.0 |

### 2.3 分支操作流程

```
功能开发流程：

1. 从develop创建功能分支
   git checkout develop
   git checkout -b feature/new-feature

2. 在功能分支上开发
   git add .
   git commit -m "feat: 实现新功能"

3. 开发完成后合并回develop
   git checkout develop
   git merge feature/new-feature
   git branch -d feature/new-feature

4. 发布时从develop创建release分支
   git checkout -b release/v0.1.0

5. release测试完成后合并到main
   git checkout main
   git merge release/v0.1.0
   git tag v0.1.0
```

## 三、Commit规范

### 3.1 Commit格式

```
格式：<type>(<scope>): <subject>

示例：
├── feat(core): 实现Rider类移动逻辑
├── fix(collision): 修复碰撞检测边界问题
├── docs(sdd): 更新需求文档
├── style(ui): 调整关卡选择界面样式
├── refactor(engine): 优化GameEngine状态管理
├── test(level): 添加关卡单元测试
├── chore(build): 配置构建脚本
```

### 3.2 Commit类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | feat(level): 添加红绿灯机制 |
| `fix` | Bug修复 | fix(rider): 修复弹回起点位置错误 |
| `docs` | 文档更新 | docs(plan): 更新开发计划 |
| `style` | 样式调整 | style(ui): 调整按钮样式 |
| `refactor` | 代码重构 | refactor(core): 优化碰撞检测算法 |
| `test` | 测试相关 | test(collision): 添加碰撞单元测试 |
| `chore` | 构建/配置 | chore(git): 配置.gitignore |

### 3.3 Commit范围

| 范围 | 说明 |
|------|------|
| `core` | 核心类 |
| `ui` | UI组件 |
| `level` | 关卡系统 |
| `collision` | 碰撞检测 |
| `engine` | 游戏引擎 |
| `audio` | 音效系统 |
| `ad` | 广告系统 |
| `build` | 构建配置 |
| `docs` | 文档 |
| `sdd` | SDD文档 |

## 四、Tag规范

### 4.1 Tag命名

```
格式：vVERSION

示例：
├── v0.1.0-alpha   # MVP完成
├── v0.2.0-beta    # 第一轮迭代
├── v1.0.0         # 正式发布
├── v1.1.0         # 功能更新
```

### 4.2 Tag创建流程

```bash
# 创建Tag
git tag -a v0.1.0-alpha -m "MVP版本完成"

# 推送Tag到远程
git push origin v0.1.0-alpha

# 推送所有Tag
git push origin --tags
```
